## 1. System Overview

**EMA DocFlow** is a procurement request and document approval system designed for internal organizational use. It replaces paper-based approval workflows with a digital system where:

- **Requesters** create procurement requests and upload supporting documents (PDFs, Word files).
- **Approvers** review and either sign (approve) or return (reject with comments) each request.
- **Admins** manage user accounts, assign roles, and monitor all activity via audit logs.

The system enforces a **sequential approval chain** — approvers must sign in order (Step 1 → Step 2 → Step 3), and a request is only fully approved when every approver has signed.

### Why This Matters

| Paper-Based Process | EMA DocFlow |
|---|---|
| Physical documents circulated by hand | Digital uploads accessible anywhere |
| No tracking of who signed when | Full audit trail with timestamps |
| Signatures can be forged | Digital signatures tied to authenticated accounts |
| No visibility into approval status | Real-time status tracking |
| Documents can be lost | Persistent cloud storage |

## 4. Database Schema

### Tables Overview

| Table | Purpose | Row Count Estimate |
|---|---|---|
| `profiles` | User display names, emails, departments | One per user |
| `user_roles` | Role assignments (admin, approver, user) | One+ per user |
| `procurement_requests` | Main request records | Grows over time |
| `approval_steps` | Individual approval steps per request | 1-5 per request |
| `request_files` | Uploaded document metadata | 1-3 per request |
| `comments` | Return/rejection reasons | 0-2 per request |
| `audit_logs` | Security audit trail | Every significant action |
| `qr_signing_tokens` | Temporary QR codes for phone signing | Short-lived (15 min) |

### Entity Relationships

```
profiles (1) ←──── (N) user_roles
    │
    │ (requester_id)
    ▼
procurement_requests (1) ←──── (N) approval_steps
    │                              │
    │                              │ (step_id)
    │                              ▼
    ├──── (N) request_files    qr_signing_tokens
    │
    ├──── (N) comments
    │
    └──── (N) audit_logs
```

### Key Design Decisions

1. **Roles in a Separate Table**: User roles are stored in `user_roles`, NOT on the `profiles` table. This prevents privilege escalation attacks where a user could modify their own profile to gain admin access.

2. **Auto-generated Short IDs**: Each request gets a human-readable ID like `REQ-0001` via a PostgreSQL sequence + trigger, making it easy to reference in emails and conversations.

3. **Approval Status Propagation**: A database trigger (`update_request_status`) automatically updates the parent request's status whenever an approval step changes — ensuring the status is always accurate.

4. **QR Tokens with Expiry**: QR signing tokens expire after 15 minutes and can only be used once. This prevents replay attacks where someone reuses an old QR code.

### Role-Based Access Control (RBAC)

| Role | Can Create Requests | Can Approve | Can Manage Users | Can View Audit Logs |
|---|---|---|---|---|
| **User** | ✅ | ❌ | ❌ | ❌ |
| **Approver** | ✅ | ✅ (assigned only) | ❌ | ❌ |
| **Admin** | ✅ | ✅ | ✅ | ✅ |

### Where Authorization is Enforced

1. **Frontend**: Route guards (`ProtectedRoute`), conditional UI rendering (admin menu items)
2. **Database**: RLS policies on every table (the **primary** security layer)
3. **Edge Functions**: Service-role validation for admin operations

> **Important**: The frontend checks are for UX convenience. Even if someone bypasses the UI, the database RLS policies prevent unauthorized data access.

## 6. Security Architecture

### Defense in Depth

```
Layer 1: Input Validation (Zod schemas)
    ↓ Sanitized, type-safe data
Layer 2: Authentication (Supabase Auth JWT)
    ↓ Verified user identity
Layer 3: Authorization (RLS policies)
    ↓ Database-enforced access control
Layer 4: Audit Trail (audit_logs table)
    ↓ Immutable record of all actions
```

### Row Level Security (RLS) — How It Works

Every database query passes through RLS policies. Example for `procurement_requests`:

```sql
-- Anyone logged in can READ all requests
CREATE POLICY "Authenticated users can read requests"
  ON procurement_requests FOR SELECT
  TO authenticated
  USING (true);

-- Only the requester can CREATE their own requests
CREATE POLICY "Users can create requests"
  ON procurement_requests FOR INSERT
  TO authenticated
  WITH CHECK (requester_id = auth.uid());

-- Only the requester or an admin can UPDATE a request
CREATE POLICY "Requesters can update own requests"
  ON procurement_requests FOR UPDATE
  TO authenticated
  USING (requester_id = auth.uid() OR has_role(auth.uid(), 'admin'));
```

### Input Validation with Zod

Every form in the application validates input before submission:

```typescript
// Example: Login form validation
const loginSchema = z.object({
  email: z.string().trim().email('Please enter a valid email address').max(255),
  password: z.string().min(6, 'Password must be at least 6 characters').max(128),
});
```

This prevents:
- SQL injection (data is parameterized by Supabase SDK, validated by Zod)
- XSS attacks (React auto-escapes output, Zod validates input)
- Buffer overflow attempts (max length limits)

### Audit Logging

Every significant action is logged with:
- **Who**: User ID
- **What**: Action type (CREATE_REQUEST, SIGN_STEP, RETURN_STEP, etc.)
- **When**: Timestamp
- **Where**: IP address, User Agent
- **Context**: Resource type, resource ID, additional details (JSON)

## 8. Key Features

### 8.1 Sequential Approval Chain

Approvers must sign in order. Step 2 cannot sign until Step 1 is approved. This is enforced in the `currentUserStep` logic:

```typescript
// A step is actionable only if:
// 1. It's assigned to the current user
// 2. Its status is WAITING
// 3. It's the first step, OR the previous step is APPROVED
const currentUserStep = steps.find(
  (step) =>
    step.approver_id === user.id &&
    step.status === 'WAITING' &&
    (step.order_index === 0 || steps[step.order_index - 1]?.status === 'APPROVED')
);
```

### 8.2 Real-Time Updates

When an approver signs a document, other users viewing the same request see the update instantly — no page refresh needed. This uses Supabase Realtime (PostgreSQL NOTIFY under the hood).

### 8.3 PDF Download with Embedded Signatures

Once all approvers have signed, the system can generate a PDF with all signatures visually embedded at their designated positions using the `pdf-lib` library.

### 8.4 Inline Document Viewing

Request documents are never opened through raw storage URLs. `RequestDetail.tsx` calls the `inline-request-file` backend function, converts the returned file bytes into a browser `Blob`, and passes only the resulting `blob:` URL into the PDF viewer. This prevents accidental downloads from storage headers while preserving explicit download buttons for generated signed PDFs.

### 8.5 Admin Dashboard

Admins can:
- View all registered users with their roles
- Change any user's role (User ↔ Approver ↔ Admin)
- Create new user accounts (via Edge Function with service role)
- Search and browse the complete audit trail

## 11. Defense Guide — Explaining to Your Supervisor

### "Why did you choose React?"

React is the most widely-used frontend framework (used by Meta, Netflix, Airbnb). It uses a component-based architecture where each piece of UI is an independent, reusable unit. TypeScript adds compile-time type checking, catching bugs before they reach users.

### "How is the data secured?"

**Three layers of security:**
1. **Input validation** (Zod) ensures no malicious data enters the system
2. **Authentication** (JWT tokens) verifies who the user is
3. **Row Level Security** (PostgreSQL policies) enforces what each user can access — this runs at the database level, so even if someone bypasses the frontend, they cannot access unauthorized data

### "What prevents someone from making themselves an admin?"

Roles are stored in a **separate `user_roles` table** with RLS policies that only allow admins to modify roles. A user cannot update their own role because the RLS policy requires the `has_role(auth.uid(), 'admin')` check to pass. The `has_role` function uses `SECURITY DEFINER` to prevent recursive policy checks.

### "What is an audit log and why does it matter?"

Every significant action (creating a request, signing a document, returning a request, creating a user) is recorded in the `audit_logs` table with: who did it, what they did, when, from which IP address, and additional context. This provides:
- **Accountability**: Every action is traceable to a specific user
- **Non-repudiation**: Users cannot deny having signed a document
- **Compliance**: Full history available for organizational audits

### "How does the QR signing work across devices?"

1. The PC generates a unique, time-limited token (expires in 15 minutes)
2. The token is stored in the database with a reference to the specific approval step
3. The QR code encodes a URL pointing to this token
4. When the phone signs, it updates the token in the database
5. The PC detects this change via a real-time WebSocket subscription and immediately shows the signature

This works because both devices communicate through the **database** (not directly to each other), so it works even on different networks.

### "Can this scale to many users?"

Yes. The backend is **serverless** — it automatically scales with demand. PostgreSQL handles concurrent queries efficiently, and the CDN serves the frontend globally. The free tier supports hundreds of users easily.

### "What happens if the server goes down?"

Lovable Cloud is built on infrastructure with high availability. Database backups are automatic. If a user loses connection temporarily, React Query will retry failed requests automatically when connectivity is restored.
